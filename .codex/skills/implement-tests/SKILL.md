# Implement Tests

## 基本方針

実装の詳細ではなく、観測可能な振る舞いをテストする。

テストを書く前に、利用者または呼び出し元から見える公開された振る舞いを特定する。優先して検証する出口は次の 3 つ。

- 戻り値
- 公開 API、画面表示、公開アクセサなどを通して観測できる状態変化
- アプリケーション外から観測できる管理外依存への呼び出し

private メソッド、private フィールド、内部ヘルパーの呼び出し、中間変数、保存形式、仕様に含まれない呼び出し順は検証しない。それらが公開契約の一部である場合だけ例外として扱う。

## 実装手順

1. 検証する振る舞いを明確にする。
   - 入力、操作、外から観測できる期待結果を言語化する。
   - 期待仕様や公開契約が不明な場合は、推測で進めず依頼者へ確認する。
   - 例示コードを書く必要があるが具体的な値や契約が未確定の場合は、その値が仮定であることを明記し、確定前に確認すべき項目を併記する。
   - 未確定の金額、ID、ラベル名、status、戻り値 shape、メール本文などは実値のように断定せず、`expectedTotal`、`expectedLabels`、`expectedStatus`、`expectedEmailPayload` などの named placeholder やコメントで表す。もっともらしい文字列リテラルを仮値として置かない。
   - 依頼文の中で公開契約が不足または矛盾している場合は、「確認事項」と「現時点で書けるテスト方針・雛形」を分ける。矛盾している項目は、確定するまで必須アサーションとして書かず、条件付きコメントに留める。

2. 検証する出口を選ぶ。
   - 結果として表れる振る舞いは戻り値で検証する。
   - 状態変化として表れる振る舞いは公開状態、画面表示、公開アクセサで検証する。
   - 外部副作用として表れる振る舞いは、管理外依存に限って呼び出しを検証する。

3. 仕様に合わせてアサーションの強さを決める。
   - 順序に意味がないなら順序を仮定しない。
   - 一部だけが仕様なら完全一致を要求しない。
   - 必須の振る舞いが欠けても通るほど緩めない。

4. モックは必要最小限にする。
   - 戻り値、公開状態、画面表示、実データに近い値で検証できるならモックを避ける。
   - メール、決済 API、メッセージバス、Webhook、分析イベントなど管理外依存の副作用はモックまたは fake で検証してよい。
   - 自前 DB、内部 repository、cache など管理下依存のメソッド呼び出しをユニットテストで検証しない。

5. 仕上げに壊れやすさを確認する。
   - 本番コードを無害にリファクタリングしただけでテストが壊れるかを考える。
   - 壊れる可能性が高い場合は、公開された結果または安定した契約にアサーションを寄せる。

## 過剰仕様のチェック

次のパターンは過剰仕様の可能性が高い。

- private または内部メソッドを spy している。
- private フィールドを見るためだけに `any` へ cast している。
- 内部ヘルパーが呼ばれたことを検証している。
- 仕様に含まれない呼び出し順を検証している。
- アプリが所有する DB や repository のメソッド呼び出しをユニットテストで検証している。
- 順序に意味がない配列で厳密な順序を期待している。
- 重要なのが一部だけの文言に対して完全一致を期待している。

代わりに次の形へ寄せる。

- 公開メソッドの戻り値を検証する。
- 画面表示、公開アクセサ、公開 API の結果を検証する。
- 永続化そのものが仕様なら、ユニットテストではなく統合テストで実物または代表的な代替を使って検証する。
- 管理外依存への外部副作用は mock または fake で検証する。

## 依存境界の判断

プロセス外依存への呼び出しを検証する前に、管理下依存か管理外依存かを分類する。

管理下依存は、このアプリケーションが管理し、外部利用者や他システムから直接観測されない依存。例: 自前 DB、内部 repository、cache。ユニットテストでは「そのメソッドが呼ばれたこと」を検証せず、公開結果で検証する。永続化の正しさが重要な場合は統合テストで扱う。

管理外依存は、外部利用者や他システムから観測できる依存。例: SMTP やメール送信、決済 API、メッセージバス、Webhook、分析イベント、通知送信。これらへの呼び出しは振る舞いの一部になり得るため、副作用が契約である場合は mock または fake で検証してよい。

## アサーションの厳密さ

仕様が求める強さに合わせる。

- 順序が仕様なら正確な順序を検証する。
- 順序は不要だが要素集合が固定なら、ソート比較や set としての比較を使う。
- 必須要素が含まれていればよく、余分な要素が許されるなら `arrayContaining` などの部分一致を使う。
- 文言の一部だけが重要なら `toContain` などの部分一致を使う。
- 完全な形が仕様なら full shape を検証する。

過剰仕様と過小仕様の両方を避ける。必須の振る舞いが変わったら失敗し、実装方法だけが変わったなら通り続けるテストにする。

## private ロジックの扱い

テストのためだけに private メソッドを公開しない。

private ロジックを直接テストしたくなる場合は、それが独立したドメイン関心かを判断する。独立した関心なら public な関数、クラス、モジュールへ切り出し、その公開契約をテストする。独立した関心でないなら、元の公開された振る舞いを通して検証する。

## テスト名

テスト名は、内部手順ではなく「前提条件と期待される振る舞い」が分かる形にする。

推奨:

```ts
it("商品を追加すると税込み合計に反映される", () => {
  // ...
});
```

非推奨:

```ts
it("calcSubtotal が呼ばれる", () => {
  // ...
});
```
